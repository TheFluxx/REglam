# Регламент одиночной фронтенд-разработки сайта для выявления UI/UX ошибок

## 1. Назначение
Документ задает единый стандарт для локальной разработки статического сайта одним разработчиком с последующим разбором ошибок другими участниками.

## 2. Область и ограничения
1. Только фронтенд, без backend, БД и хостинга.
2. Локальный запуск (`index.html` или статический сервер).
3. Обязательный стек: `HTML/CSS/JS`.
4. Итоговая модель приемки: `Pass/Fail`.

## 3. Обязательные артефакты
```text
project_root/
├── index.html
├── assets/css/styles.css
├── assets/js/main.js
├── assets/img/
├── docs/00_reglament_single_dev.md
├── docs/01_spec_scope.md
├── docs/02_known_gaps.md
├── docs/03_review_log.md
├── docs/04_decisions.md
├── docs/05_final_checklist.md
└── README.md
```

## 4. Минимальные требования к сайту
1. Минимум 4 секции: `Hero`, контент, форма/интерактив, футер.
2. Минимум 2 пользовательских сценария (основной и альтернативный).
3. Адаптивность: 360px+ и 1280px+.
4. Базовая доступность: заголовки, фокус, контраст, `alt`.
5. Без критических ошибок в консоли в основном сценарии.

## 5. Версионирование и freeze
1. Контрольные итерации: `v01`, `v02`, `v03`.
2. На каждой итерации фиксируются изменения, незавершенности и риски.
3. Перед ревью версия замораживается (freeze).
4. Любая правка после freeze обязана иметь запись в `docs/04_decisions.md`.

## 6. Правила фиксации незавершенных деталей
Каждая карточка в `docs/02_known_gaps.md` содержит:
1. `ID` (`GAP-XX`)
2. Описание
3. Причину незавершенности
4. Ожидаемое корректное поведение
5. Риск (`Low/Medium/High`)
6. Статус (`Open/Discussed/Resolved/Rejected`)

## 7. Формат журнала ревью
`docs/03_review_log.md` использует поля:
`Issue ID`, `Категория`, `Серьезность`, `Где найдено`, `Шаги воспроизведения`, `Ожидаемо`, `Фактически`, `Комментарий разработчика`, `Решение`, `Статус`.

## 8. Критерии Pass/Fail
`Pass` ставится только если:
1. Нет открытых `Critical`.
2. Все `Major` имеют решение (`Fixed` или `Accepted with rationale`).
3. Все обязательные артефакты присутствуют.
4. Заполнены `docs/03_review_log.md`, `docs/04_decisions.md`, `docs/05_final_checklist.md`.
5. Локальный запуск проходит по `README.md`.

Иначе итог: `Fail (на доработку)`.

## 9. Типовые нарушения
1. Нет списка незавершенных деталей.
2. Замечания обсуждаются устно без фиксации.
3. Нет шагов воспроизведения.
4. Фикс сделан без решения/обоснования.
5. Нет покрытия 2 пользовательских сценариев.
6. Проверка только на одном разрешении.
7. Нет зафиксированной freeze-версии.
