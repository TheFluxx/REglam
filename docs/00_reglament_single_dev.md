# Регламент одиночной фронтенд-разработки для UI/UX ревью (v2.0)

## 1. Назначение
Документ задает усложненный, но умеренно строгий стандарт локальной разработки статического сайта одним разработчиком с формализованным аудитом.

## 2. Область и ограничения
1. Только фронтенд, без backend, БД и хостинга.
2. Локальный запуск (`index.html` или статический сервер).
3. Обязательный стек: `HTML/CSS/JS`.
4. Приемка только `Pass/Fail`.
5. Количество ревьюеров не фиксируется, но все замечания фиксируются в документах.

## 4. Минимальные требования к сайту
1. Минимум 4 секции: `Hero`, контентный блок, форма/интерактив, футер.
2. Минимум 2 пользовательских сценария (`основной`, `альтернативный`).
3. Для каждого сценария обязательны `happy-path`, `invalid-path`, `recovery-path`.
4. Адаптивность: `360px+` и `1280px+`.
5. A11y минимум: клавиатурная проходимость, фокус, семантика заголовков, осмысленные `alt`.
6. Performance минимум: без критических ошибок консоли, без тяжелых неиспользуемых ассетов.
7. Content минимум: единый тон текста, без `lorem/todo/tbd` в UI.

## 5. Версионирование и freeze
1. Фиксированные этапы: `v01`, `v02`, `v03`, `v04`, `RC`, `Final`.
2. Значение этапов:
- `v01`: каркас навигации и базовая структура.
- `v02`: UX-проходы и обработка ошибок.
- `v03`: A11y и контентная полировка.
- `v04`: стабилизация и закрытие Major.
- `RC`: freeze-кандидат.
- `Final`: только после прохождения quality-gates.
3. После `RC` любые правки фиксируются отдельной записью в `docs/04_decisions.md`.
4. Каждая итерация обязана иметь запись в `docs/08_release_notes.md`.
5. Неполная запись release notes делает итерацию недействительной.

## 6. Формат фиксации незавершенностей и дефектов
1. `docs/02_known_gaps.md` использует формат `GapCardV2`.
2. `docs/03_review_log.md` использует формат `ReviewIssueV2`.
3. `docs/04_decisions.md` использует формат `DecisionRecordV2`.
4. `docs/05_final_checklist.md` использует формат `GateResult`.

## 7. Quality-gates (блокеры Pass/Fail)
`Pass` возможен только при одновременном выполнении:
1. Defects gate:
- `Critical/Open = 0`.
- `Major/Open = 0` или `Accepted with rationale` c `Expiry`.
2. UX gate:
- пройдены 2 сценария end-to-end;
- подтверждены `happy`, `invalid`, `recovery`.
3. A11y gate:
- клавиатурная проходимость основных интерактивных элементов;
- видимый фокус;
- корректная heading-иерархия;
- осмысленные `alt`.
4. Performance gate:
- нет деградации первой отрисовки между `v04` и `RC`;
- нет тяжелых неиспользуемых ассетов;
- нет критических ошибок в консоли.
5. Content gate:
- единый стиль микрокопирайта;
- нет технических заглушек в UI;
- сообщения ошибок формулируют действие.

## 8. Модель итоговой приемки
1. `Pass`: все gates = `PASS`, обязательные документы заполнены, `RC` задокументирован.
2. `Fail`: хотя бы один блокирующий критерий не выполнен.
3. `Fail (Conditional)`: допускается только при документированном принятии риска с `Expiry`.

## 9. Типовые нарушения
1. Нет трассировки между `GAP`, `Issue`, `Decision`, `Risk`.
2. Нет заполненной тест-матрицы.
3. Нет релизных заметок по одной или более итерации.
4. Обсуждение проблем только устно, без фиксации.
5. Закрытие `Critical/Major` без решения и даты.
6. Попытка финальной приемки до `RC`.
